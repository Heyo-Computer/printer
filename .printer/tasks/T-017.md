## Notes

- 2026-05-01T18:20:35Z [sarocu] Plan — wire ACP (Agent Client Protocol) as a third agent transport alongside Claude and Opencode.

Context: today `printer/src/agent.rs::AgentInvocation` shells one-shot `claude --print` / `opencode run` per turn and parses stdout JSON. ACP is JSON-RPC 2.0 over a long-lived stdio child with `initialize`, `session/new`, `session/prompt`, and streaming `session/update` notifications (https://agentclientprotocol.com). Per-turn one-shot does not map cleanly — the agent process must persist across `Session::turn` calls.

Files to touch:
- `printer/src/cli.rs` — add `AgentKind::Acp` variant; add CLI flags `--acp-bin` (path) and `--acp-arg` (repeatable) on `RunArgs`/`ReviewArgs`/`ExecArgs`/`PlanArgs`/`SpecFromFollowupsArgs`. (Or read from plugin manifest — see T-018.)
- `printer/src/agent.rs` — split `AgentInvocation` so non-ACP keeps the spawn-per-turn `Command` builder, and ACP grows a new path that holds an open child + JSON-RPC reader/writer. New `AgentBackend` enum: `OneShot { argv builder }` vs `Acp { handle to long-lived client }`. `bootstrap`/`resume`/`parse_outcome` move behind this enum.
- New module `printer/src/agent/acp.rs` (or `acp_client.rs`):
  * `AcpClient` struct: owns `tokio::process::Child`, line-based JSON-RPC framing on stdin/stdout, monotonic id counter, pending-request map (`HashMap<u64, oneshot::Sender<Value>>`).
  * Tasks: reader loop (parse one JSON object per line, dispatch `result`/`error` to oneshot, dispatch notifications to a streaming sink); writer is `Mutex<ChildStdin>`.
  * Methods: `initialize()`, `session_new(cwd, mcp_servers)`, `prompt(session_id, content) -> impl Stream<Item = Update>`, `cancel(session_id)`, `shutdown()`.
  * Wrap launch through the sandbox `enter` template (same shell-quote machinery as `drivers::shell_quote_argv`) so ACP servers run inside heyvm — relevant because ACP is a long process, not per-turn.
- `printer/src/session.rs` — when backend is `Acp`, do not spawn a new process per `turn`; instead reuse the held client. The `fresh` flag still means "send `session/new`", subsequent turns send `session/prompt` against the saved session id. Map streamed `session/update` content blocks → `TurnOutcome.result_text`; usage may be absent (default to zero, with a warning the first time).
- `printer/src/run.rs`, `printer/src/review.rs`, `printer/src/exec.rs` — when `AgentKind::Acp`, lifecycle the client at the same scope where today the per-process `Command` would be spawned. For exec, one ACP client spans both run and review (matches the single-sandbox lifecycle).
- `printer/Cargo.toml` — add `tokio` features for child stdio if not already on; `serde_json` already present.
- `printer/HOOKS.md` — document `--agent acp` and the env contract.

Sub-task to split out (keeps T-017 a single PR):
- A new follow-up task covers proper streaming + cancellation + permission-mode mapping. The umbrella T-017 only needs blocking turn semantics (wait for `session/prompt` to resolve, collect all `session/update` blocks into result_text). Streaming + Ctrl-C cancellation via `session/cancel` becomes its own task.

Verification:
- `cargo build -p printer` and `cargo test -p printer agent::` (new unit tests for JSON-RPC framing + reader-loop dispatch with a mocked stdin/stdout pair).
- Round-trip integration: spawn a tiny ACP server stub from `tests/` that echoes a fixed reply, drive one turn, assert `TurnOutcome.result_text`.
- Manual: `printer run spec.md --agent acp --acp-bin "claude-code-acp-stub"` against a mock; confirm cwd and stop on `<<DONE>>`.

Dependencies: none for the transport itself. T-018 depends on this.
- 2026-05-01T18:29:11Z [sarocu] Implementation complete.

Wire-up:
- printer/src/cli.rs: AgentKind::Acp variant + --acp-bin / --acp-arg on Run/Plan/Review/Exec/SpecFromFollowups.
- printer/src/agent/{mod.rs,acp.rs}: agent.rs split into a directory module; new acp.rs holds the JSON-RPC client (line-delimited JSON-RPC 2.0 over stdio, monotonic id counter, pending oneshot map, mpsc for session/update notifications, sandbox-wrapper-aware spawn). bootstrap()/resume() panic for AgentKind::Acp — the ACP path bypasses Command construction.
- printer/src/session.rs: AgentKind::Acp dispatches to a new Session::turn_acp that lazily spawns AcpClient on first use, drives initialize → session/new, then prompt_blocking per turn. rotate() drops the client so the next turn re-spawns.
- run.rs / review.rs / exec.rs / plan.rs / spec_from_followups.rs: thread acp_bin / acp_args through AgentInvocation construction.
- printer/HOOKS.md: documents --agent acp, sandbox wrapping, and what's deferred to T-020 (token usage, streaming, cancellation, permission mode).

Verification:
- cargo build clean (no warnings).
- cargo test: 96 existing tests pass + 4 new agent::acp tests covering response dispatch, error dispatch, session/update notification routing, and content-block text collection.
- Manual smoke: 'printer run --agent acp' with no --acp-bin fails with the expected message; round-trip against /tmp/acp-stub.py confirms persistent client across turns (turn 1 fresh + session/new, turns 2/3 'resumed' on the same session id) with notification text aggregated into result_text.
