- 2026-05-02T22:05:28Z [sarocu] Plan — add `computer browse <URL>` subcommand.

Files
- computer/src/main.rs — add `Cmd::Browse { url: String }` (between `Type` and `Sleep`); dispatch to `platform::browse::run(&url)`.
- computer/src/platform/mod.rs — add `pub mod browse;` (shared, same logic on Linux + macOS).
- computer/src/platform/browse.rs (new, ~15 LOC) — `pub fn run(url: &str) -> anyhow::Result<()>` that:
  - Validates `url` parses as `http(s)://` or `file://` (reject empty / shell-metachar-only input to avoid foot-guns; do not over-validate, just bail on clearly bogus values).
  - Spawns `xdg-open <url>` on `cfg(target_os = "linux")`, `open <url>` on `cfg(target_os = "macos")`. Use `std::process::Command::new(...).arg(url).spawn()` (don't wait — these forks return immediately) and propagate the launch error with anyhow context.
- plugins/computer/skills/computer/SKILL.md + README.md — add the new verb to the command surface table and a "launch a URL for click-testing" example. (Skill copy lives in the plugin tree only; .claude/skills mirror is regenerated by plugin install.)

Verification
- `cargo build -p computer` clean.
- `cargo test -p computer` (no new tests required; the existing tree just needs to keep compiling).
- Manual: `computer browse https://example.com` opens the default browser; `computer browse ''` returns a non-zero exit with a clear error.
- `computer help` lists the new verb.

Out of scope
- No URL allow-listing / sandboxing — that's a host-level concern.
- No "wait for browser to exit" — fire-and-forget matches `xdg-open` semantics.
- 2026-05-02T22:09:13Z [sarocu] Refinement after re-reading the crate: existing pattern in computer/src/platform/mod.rs is per-OS subdirs (linux/, macos/) re-exported via cfg gates — there is no flat platform/*.rs sibling module. Two options:
  (a) follow the convention: add platform/linux/browse.rs and platform/macos/browse.rs (each ~5 LOC), then in platform/mod.rs append `browse` to the existing `pub use linux::{...}` / `pub use macos::{...}` re-export lists.
  (b) one cfg-gated platform/browse.rs at the top level, sibling to types.rs/keymap_common.rs (which are already shared, non-OS-specific).
Pick (a) — keeps the convention uniform; the `xdg-open` vs `open` split is exactly the kind of thing the per-OS layout exists for. Dispatch from main.rs becomes `platform::browse::run(&url)` either way.
