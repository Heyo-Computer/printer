- 2026-05-01T18:21:37Z [sarocu] Plan — author a concrete `plugins/poolside/` plugin that wires Poolside's CLI as an ACP agent, using the framework from T-018.

Source: https://docs.poolside.ai/cli/editor-integration#editor-integration-acp — Poolside's CLI runs an ACP server when started with the appropriate subcommand (verify the exact invocation against the docs at implementation time; the plugin manifest is just the launch glue).

Files to create:
- `plugins/poolside/printer-plugin.toml`:
    assets = ["skills"]
    [[agent]]
    kind = "acp"
    name = "poolside"
    command = "poolside"
    args = ["acp"]   # confirm against poolside docs
    # env left empty by default; users opt in via host env or shell profile.
- `plugins/poolside/skills/poolside/SKILL.md` — short brief telling the agent it is the Poolside model and to follow the user's existing project conventions; mirrors structure of `plugins/heyvm/skills/heyvm-sandbox/SKILL.md`.
- `plugins/poolside/README.md` — install instructions:
    printer add-plugin path:plugins/poolside
    printer run spec.md --agent acp:poolside
  Plus a Prerequisites section noting that the `poolside` CLI must be on $PATH (link to docs.poolside.ai).

Files to update:
- `printer/HOOKS.md` — under the new "ACP agents" section from T-018, add Poolside as the worked example (parallel to heyvm being the worked driver example).

Verification:
- `printer add-plugin path:plugins/poolside` lands a manifest under `~/.printer/plugins/poolside/` with `agent` in the ROLES column of `printer plugins`.
- `printer run --agent acp:poolside <spec>` resolves to the manifest's command and dispatches one turn end-to-end against a real `poolside` CLI install (or a stub when CI lacks the binary).
- A pre-existing user with no Poolside credentials gets a clear error from the ACP transport (surfaced from poolside's stderr), not a silent hang.
